# KELEDON Cloud OpenTelemetry Observability

## Why OpenTelemetry

KELEDON Cloud uses OpenTelemetry so tracing is standards-based and vendor-neutral.
This replaces building and maintaining a custom tracing pipeline (custom span formats,
manual exporters, and bespoke viewer tooling).

For local development, traces are exported to Jaeger through OTLP/HTTP.

- Service: `keledon-cloud`
- Exporter: `@opentelemetry/exporter-trace-otlp-http`
- Default endpoint: `http://localhost:4318/v1/traces`
- Jaeger UI: `http://localhost:16686`

## Canonical Span Model

Trace vocabulary is defined in `cloud/src/telemetry/trace-model.ts`:

- `keledon.listen`
- `keledon.transcribe`
- `keledon.vector.retrieve`
- `keledon.policy.check`
- `keledon.decide`
- `keledon.command.emit`
- `keledon.rpa.execute`
- `keledon.respond`
- `keledon.speak`

These spans map directly to the canonical loop:
`LISTEN -> TRANSCRIBE -> THINK (Cloud + Vector) -> DECIDE -> ACT (RPA) -> RESPOND -> SPEAK -> LOOP`

## What Is Instrumented In This Session

1. Telemetry bootstrap before Nest app startup
   - File: `cloud/src/telemetry/otel.ts`
   - Uses NodeSDK, OTLP exporter, and HTTP auto-instrumentation

2. Decision trace with structured evidence
    - File: `cloud/src/services/decision-engine.service.ts`
    - Span: `keledon.decide`
    - Attributes:
      - `decision.id`
      - `decision.policy_ids[]`
      - `decision.playbook_id`
      - `decision.vector_collections[]`
      - `decision.vector_doc_ids[]`
      - `decision.confidence_score`
      - `decision.type` (`RESPOND | ASK | ACT | WAIT`)
      - `model`, `token_count` (when available)

3. Policy enforcement trace
   - File: `cloud/src/services/decision-engine.service.ts`
   - Span: `keledon.policy.check`
   - Attributes: `policy.decision_id`, `policy.ids[]`, `policy.playbook_id`, `policy.applied`
   - Failure rule: if no policy applies, the decision fails

4. Vector retrieval trace (mandatory)
    - Files:
      - `cloud/src/services/decision-engine.service.ts`
      - `cloud/src/rag/rag.service.ts`
    - Span: `keledon.vector.retrieve`
    - Attributes: `vector.collection`, `topK`, `latency_ms`, `vector.doc_ids[]`
    - Failure rule: if retrieval returns zero documents in decision flow, `keledon.decide` fails

5. Command emission trace
    - File: `cloud/src/gateways/agent.gateway.ts`
    - Span: `keledon.command.emit`
    - Attributes: `command_type`, `session_id`, `decision.id` (when available)

6. Context propagation evidence
    - HTTP responses now include `x-trace-id`
    - WebSocket command payload metadata includes `trace_id` and `decision_id`
    - WebSocket inbound metadata can carry `traceparent`/`tracestate`

## Why This Prevents Fake State

- Trace IDs and spans are generated by the runtime and exported out-of-process to Jaeger.
- Decision and vector spans are tied to real code execution points, not UI status labels.
- If vector retrieval does not run, `keledon.vector.retrieve` is absent and decision flow can fail loudly.
- Browser-side decisioning cannot be hidden because decision spans are emitted only from Cloud services.

## Local Verification Commands

Start Jaeger:

```bash
docker run -d --name keledon-jaeger -p 16686:16686 -p 4318:4318 jaegertracing/all-in-one:latest
```

Run cloud (TypeScript runtime):

```bash
cd cloud
npx ts-node src/main.ts
```

Trigger API + vector retrieval:

```bash
curl -i http://localhost:3001/api
curl -H "Content-Type: application/json" -X POST http://localhost:3001/rag/retrieve -d '{"query":"How do we resolve this ticket?","sessionId":"session-otel-1","companyId":"keledon","maxResults":2}'
```

Check Jaeger API:

```bash
curl "http://localhost:16686/api/traces?service=keledon-cloud&operation=keledon.vector.retrieve&limit=5"
```
